**Result:** `guess_function.py` (v1) + `make_data.py`

Key design decisions:

- Simple evolutionary search: random population → score by MSE → keep top 20%
  → mutate → repeat

**Result:** `guess_function.py` (v2)

Key changes:

- Changed fitness from pure MSE to `MSE + λ · size` (MDL)
- Added `COMPLEXITY_WEIGHT = 0.001` (λ) — cost of each extra node

**Problem noticed:** Search found `cos(π/2 + x²)` rather than `sin(x²)` —
algebraically identical, but 14 nodes vs 7. Root cause: syntactic size ≠
semantic/algebraic complexity. Needs a simplifier to canonicalise before
scoring.

**Result:** `guess_function_pysr.py` (v1)

Key design:
- `PySRRegressor` with same primitive set as hand-rolled version

## Turn 7 — Why x0?

**User:**

> What is x0? Why is this not just in terms of x? Is it because there could be
> multiple variables?

**Answer:** Yes exactly. PySR is designed for the multivariate case; it takes a
feature matrix `X` with shape `(n_samples, n_features)` and names columns
`x0, x1, x2, ...`. With a single variable you get `x0`. Fixed by passing
`variable_names=["x"]` to `model.fit()`.
